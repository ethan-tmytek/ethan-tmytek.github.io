## 一、先講錢，再講工具

導入 AI 工具的提案最常死在一個地方：**把工具當主角，把痛點當配角**。

正確的順序是反過來的——先指認一個正在燒錢的具體問題，再讓工具去解。如果這個問題沒有 AI 也該解，那就更值得導入；AI 只是把「該做但沒人有時間做」的事情變成做得完。

軟硬整合團隊有一個經典的、反覆出現、但很少被系統性根治的問題：**版本黑洞**。本文以這個痛點為錨點，串接其他幾個高頻痛點，提出一份具體的 90 天試點計畫。

---

## 二、主痛點：「我沒改 code 啊」

每位帶過軟硬整合團隊的 Leader 都聽過這句話。典型現場：

1. QA 回報：APP 顯示的 ADC 值跟示波器對不上
2. APP 工程師：「我沒改 code 啊，昨天還好好的」
3. 查兩小時 → APP 連到的是舊版 API
4. 再查一小時 → FW 是同事昨天燒的 debug 版本
5. 再查半小時 → FPGA bitstream 是上週版本
6. **真正的 bug 不在程式碼，在組合**

這不是工程師不細心，是結構性問題：

| 因素 | 為什麼難解 |
|---|---|
| 多層次組件 | FW / FPGA / API / APP 各有獨立 build 與 release 節奏 |
| 多套硬體 | 每位工程師桌上的 DUT 燒著不同版本，無從追蹤 |
| 燒錄是手動行為 | 燒一次 FW、load 一次 bitstream，往往沒留紀錄 |
| 口頭協作 | 「我幫你燒了新版」這種對話從不進 ticket |
| 沒有 single source of truth | 沒有任何地方能一鍵告訴你「現在這台機器跑的是什麼組合」 |

更糟的是**團隊對測試結果失去信心**：跑出來的 bug 是真 bug，還是版本不對？沒人敢拍胸脯。這個信心成本，比工時更難量化、也更貴。

**怎麼量這個痛點**：撈過去 3 個月 bug ticket，標記「最後發現是版本問題」的比例，乘上平均工時。一個 20 人團隊跑下來，年度損失常落在 80–150 個工程人天。這個數字是要回頭實算的，但通常都會大於 Leader 原本的直覺。

## 三、配套痛點：版本黑洞之外，還有什麼在燒時間

只解一個痛點不值得導入新工具。同樣的工具，能順帶解掉以下幾個高頻痛點，才會有複利效應：

| 痛點 | 具體情境 | AI 工具能加的價值 |
|---|---|---|
| **跨層 Bug 追蹤** | Bug 出現在 GUI，根因可能在 FW 暫存器設定。需來回切換三層 codebase | 同時讀 FW / API / APP 三層程式碼，協助快速定位根因 |
| **Firmware 除錯週期** | 改 FW → 燒錄 → 跑儀器 → 看結果，一輪 15–30 分鐘，常常改錯方向重來 | 先做靜態分析與 hypothesis ranking，減少盲改次數 |
| **儀器自動化腳本** | 每個驗證情境都要寫 Python / LabVIEW 腳本驅動儀器與收資料，重複度高但細節各異 | 擅長產生樣板化腳本，工程師只需修改關鍵參數 |
| **通訊協定一致性** | FW 改了通訊欄位，API 與 APP 端常常忘了同步更新，整合測試 fail | 一次掃過三層 codebase，列出需同步修改的位置 |
| **新人 Onboarding** | 新人要看懂 FW + API + APP 三層架構，平均上手 1–2 個月 | 即時回答「這個函式在哪用到」「為什麼這樣寫」，縮短 ramp-up |
| **Code Review 負擔** | Senior 工程師大量時間花在 review，常常只能挑大問題看 | 做「第零次 review」，先過濾風格、明顯 bug、缺測試 |

注意這六個痛點與「版本黑洞」的共同特徵：**跨層、跨檔案、重複度高、需要時間寫工具但沒人有時間寫**。這是 AI 編程工具的甜蜜點。

## 四、為什麼是 Claude Code

選 Claude Code 而不是其他工具的理由很實際：

- **Terminal 原生、agentic**：FW 工程師大量在 terminal、Makefile、build 系統間切換，比 IDE 內補全工具的姿勢更貼合。
- **跨層、跨檔案任務**：本節列出的所有痛點都跨多層 codebase，需要工具能讀懂整個 repo，而不是只看當前游標附近。
- **可程式化、可整合 CI**：能用 hooks 與 MCP 客製，長期內化為團隊資產，不是個人玩具。
- **企業條款清楚**：API 有不訓練資料的選項，採購時與 IT、Legal 確認即可。

這份提案沒有要全面取代 Copilot / Cursor。試點期間若工程師原本已用其他工具，不強制下架。聚焦 Claude Code 是因為它的特性最匹配「FW + 整合測試」這個工作型態。

## 五、解法設計：以版本黑洞為主軸，順勢解其他痛點

### 主軸：版本一致性的三層解法

**Level 1：版本資訊的自動收集**

用 Claude Code 寫一個「DUT 版本快照」工具，一鍵收集 FW 版本、FPGA bitstream rev、API server 版本、APP 版本、git commit hash、燒錄時間戳，輸出 YAML/JSON。附在每張 bug ticket 上。

關鍵：不是 AI 自己跑去檢查，而是 AI 幫你**快速寫出**這個工具。團隊內沒人有完整時間從頭寫，給 Claude Code 兩天可以產出可用版本。

**Level 2：相容性矩陣**

維護一份 `compatibility-matrix.yaml`：

```yaml
release_2024_Q4:
  firmware: ">=1.4.2, <1.5.0"
  fpga_bitstream: "rev_C"
  api: ">=2.1.0"
  app: ">=3.0.0"
  known_incompatible:
    - "FW 1.4.1 + FPGA rev_C → ADC 讀值偏移"
```

Claude Code 從 commit history 與 release notes 抽取已知相容性、收到版本快照後比對矩陣、發現新不相容組合時協助更新。

**Level 3：CI / Pre-flight Check**

每次燒錄自動產生紀錄、bug ticket 模板強制附版本快照、PR opened 時自動檢查跨層相容性風險、release 前 pre-flight check 自動驗證。

### 副軸：順勢解掉的工作流程

**跨層 Bug 追蹤**（接續主軸 Level 1）：版本快照確認組合是合法的 → 才進入真正的根因分析。把 stack trace、reproduction steps、相關檔案路徑丟給 Claude Code，**先請它分析根因，不要先寫 fix**。人類確認方向後才請它寫 fix + 測試。

**通訊協定一致性**（與主軸 Level 2 共用機制）：通訊欄位本質就是一種「跨層相容性」。把 protocol 定義納入相容性矩陣，FW 改欄位時自動掃描 API 與 APP 端需同步修改的位置。

**Firmware 除錯週期**：在燒錄前先讓 Claude Code 對 diff 做靜態分析，標出可疑邏輯與遺漏的邊界條件。減少「燒一次才知道方向錯」的次數。

**儀器自動化腳本**：把儀器 SDK 文件、既有腳本範例放進 CLAUDE.md，明確標記儀器型號與 SDK 版本。新增驗證情境時，描述需求 → Claude 產草稿 → 工程師補 edge case → 小規模驗證 → 上線。

**新人 Onboarding**：CLAUDE.md 寫得清楚，新人可直接用 Claude Code 在 repo 內問「這個函式為什麼這樣寫」「這個模組與哪些模組互動」。Senior 不需要重複講同樣的事。

**Code Review 負擔**：PR opened → CI hook 觸發 Claude review → 留 inline comment（風格、明顯 bug、缺測試、安全 smell）→ 作者標註「已處理」→ 人類 reviewer 只看標註好的 PR。AI 是「第零次 review」，不是取代人類。把 review prompt 寫在 `.claude/review-prompt.md`，全隊 iterate。

**重點**：就算 90 天後決定不續用 Claude Code，這些成果——快照工具、相容性矩陣、CI hook、CLAUDE.md、review prompt——都會留下來。這降低了試點的決策風險。不是在賭 AI，是在賭「整理開發流程」這件事值不值得做，AI 只是加速器。

## 六、90 天計畫

### Phase 1：Pilot（Day 1–30）—— 把痛點具體化

- 4–6 位自願工程師，含 1 位 Senior，涵蓋 FW / FPGA / API / APP
- **Week 1**：工具安裝、IT 確認資安、起草 CLAUDE.md、**撈舊 ticket 量化版本痛點 baseline**
- **Week 2–3**：完成 Level 1 版本快照工具；每位試點成員至少在「跨層 bug 追蹤、儀器腳本、code review 輔助」三類任務中各試一次
- **Week 4**：第一次回顧，產出 Phase 1 報告

### Phase 2：Validate（Day 31–60）—— 建立機制

- 擴大到 8–10 人
- 建立 `compatibility-matrix.yaml`，從歷史 release notes 抽取初版
- 版本快照整合進 bug ticket 模板（強制欄位）
- CI hook 上線：Claude Code 自動跑 PR review + 跨層相容性檢查
- 新人 onboarding 流程納入 Claude Code 使用指引
- **Day 60 中期決策點**：版本相關 ticket 是否下降？PR cycle time 趨勢？不對就停。

### Phase 3：Scale Decision（Day 61–90）—— 自動化與決策

- 燒錄紀錄自動化、release pre-flight check 上線
- 儀器腳本模板庫收斂成內部 playbook
- 比對 90 天前 vs 後的關鍵指標
- 三選一：全面推廣 / 部分採用（如 FW 組先用）/ 停損

## 七、KPI

**主指標——版本黑洞（必須改善才算成功）**：

| 指標 | 量法 | 90 天目標 |
|---|---|---|
| 版本相關 bug ticket 比例 | 撈過去 3 個月 ticket 標記 | 下降 50% |
| 版本相關 ticket 平均解決時間 | ticket 系統時間戳 | 下降 50% |
| 「我沒改 code 啊」事件數 | 試點成員每週自評 | 從每週 N 起 → 趨近 0 |
| DUT 版本快照覆蓋率 | 試點 DUT 比例 | 100% |

**配套指標——其他痛點**：

| 指標 | 量法 | 目標 |
|---|---|---|
| PR cycle time | Git 平台 | 下降 20% |
| Code review 第一輪回覆時間 | PR 工具 | AI review 1 小時內，人類 review 24 小時內 |
| 新人到第一個 merged PR | HR / Lead 紀錄 | 有新人時量測 |
| 儀器腳本撰寫時間 | 試點成員自評 | 下降 30% |

**領先指標**：週活躍使用率 ≥ 70%、滿意度 ≥ 3.5/5、成功案例 ≥ 10 個。

**不採用的指標**：程式碼產出量、AI 採用率本身、「節省的工時」單一誇大數字——這些是行銷話術，不是管理指標。

## 八、風險與停損

| 風險 | 緩解 |
|---|---|
| 原始碼資安疑慮 | 採購時確認企業條款；高敏感 FW 模組先不接 AI |
| AI 產生的工具有 bug 反而誤導 | 試點期人工 double check；工具納入 code review |
| 團隊不買單 | 自願制、不列績效、Senior 先試 |
| FW / FPGA 特性硬體限制 | CLAUDE.md 明確標記型號與手冊位置；初期聚焦軟體層 |
| Day 60 KPI 不達標 | 主動暫停回報，省下後 30 天成本 |

**停損點要明講**：60 天時若版本相關 ticket 數沒有任何下降趨勢，主動暫停。這比承諾「一定成功」有說服力，因為它顯示提案者想清楚了。

## 九、變革管理：幾句要先講白的話

導入 AI 工具最大的失敗原因不是技術，是人。

- **「AI 寫的 code 不能信」** → 訂原則：AI 是初稿產生器，人類 reviewer 與 merger 負責。
- **「用 AI 寫算我寫的嗎」** → 誰送 PR 誰具名，AI 是工具不是作者。
- **「我會被取代」** → Leader 公開表態：節省的時間用來做更有意思的事，不是裁員依據。
- **「我試過很爛」** → 私下了解用哪個工具、什麼任務。多半是任務選錯或工具選錯。
- **沉默式不用** → 1-on-1 私下聊；公開會議挖不出真原因。

明確的紅線：
- 不把客戶機密、未公開演算法、API key 貼進 prompt
- 不讓 AI 自動 merge PR
- 不用 AI 產出的 code 直接燒進量產 FW 而未經測試
- 不用「AI 寫的」當 bug 藉口

## 十、給 Leader 的三個本週行動

讀完這篇，這週可以做的事：

1. **撈一張真實的 bug ticket**——那種「我沒改 code 啊」最後查出來是版本問題的。開會時拿出來，比任何投影片有說服力。
2. **粗估痛點成本**：過去 30 天版本相關事件數 × 平均工時 × 時薪。一個數字就好，但要是真實數字。
3. **私下找 1 位 FW + 1 位 APP 同事**問：「如果有一個工具能一鍵告訴你這台 DUT 現在跑什麼版本組合，你會想用嗎？」眼睛亮起來的就是你的種子使用者。

## 十一、結語

導入 AI 工具不該是一個信仰問題，而是一個工程問題。

挑一個正在燒錢的痛點當主軸，順手解掉幾個高頻配套痛點，做一個 90 天的試點，設一個明確的停損點，量幾個可驗證的指標。成就成，不成就停。團隊學到的工作流程經驗，不會因為工具換了就消失。

版本黑洞值得解決，無論最後選的工具是不是 Claude Code。AI 只是讓「該做但沒人有時間做」這件事，第一次變得做得完。

幾個整合的設計決策說明一下，方便你判斷要不要再調：

**為什麼版本黑洞放主軸、其他六個放配套**：如果六個痛點並列，整篇會散，老闆看完不知道重點在哪。版本黑洞最容易量化、最有即時 ROI、最能在 30 天內看到效果——適合當「第一張多米諾骨牌」。其他六個放在 Level 2-3 與副軸描述，顯示「這不是單點解，是系統性升級」。

**Phase 1 的設計刻意保留彈性**：Week 2–3 同時讓試點成員碰版本工具、跨層 bug、儀器腳本、code review。這樣 Day 30 報告會有多個面向的數據，不會只賭版本黑洞一個指標。

**KPI 分主指標與配套指標**：主指標必須達標才算成功，配套指標達不達標都當 bonus。這個結構讓你 Day 90 報告好寫——主指標達標就推廣，沒達標但配套指標亮眼就考慮 Phase 2。
