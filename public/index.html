Day 1｜TMYTEK 的硬體專案執行心法：從共同語言開始 系列：《TMYTEK 的硬體專案執行心法：從 Kickoff 到量產的工程治理》 適讀：PJM / EPM / MPM 團隊，以及與我們協作的 BD / SCM / R&D 同仁 我們是一家硬體為主、走完整 EVT/DVT/PVT 流程、以 ODM/EMS 代工量產的軟硬整合公司。這個產業有它自己的物理法則：一塊板子打樣兩週是日曆時間、一次認證測試是外部資源、一個新機種從 Concept 到量產是 9 到 18 個月。我們不能用純軟體公司「再加一個衝刺修一下」的思維來管理硬體專案。 要把這件事做好，第一步不是工具，是共同語言。讓 PJM、R&D、BD、SCM 在同一張表上能看懂彼此在做什麼、資源怎麼分配、戰略投資是否合理。這就是這篇文章要建立的基礎。 一、Investment Category：八大投入類別 我們把 R&D 的所有工作分成 8 個投入類別。每一個 Epic（後面會解釋這是什麼）都必須掛上一個類別。這 8 類是這樣設計的： # 類別 說明 1 新硬體產品 / 新機種 新 SKU、新硬體架構，含其必要的韌體與配套 APP/雲端 2 既有產品迭代 已上市產品的功能更新、效能優化、體驗改善 3 品質 / Bug / RMA 軟硬韌的 bug、量產線品質事件、客訴根因、產線異常 4 驗證 / 認證 / 合規 EVT/DVT/PVT 各階段測試、各國法規認證、安規、EMC、可靠度測試 5 製造 / NPI / 量產導入 產線治具、產測程式、SOP 撰寫、與代工廠對接、試產支援 6 平台 / 工具鏈 / 自動化 共用韌體 SDK、雲端後端平台、自動化測試平台、CI/CD、開發工具 7 技術預研 / 先期研究 新技術評估、新晶片導入評估、新製程或新材料試作、Reference Design 研究 8 營運支援 / KTLO OTA 維運、客戶技術支援升級、產線即時支援、Field Issue 處理 為什麼是這 8 類，不是 5 類也不是 15 類 純軟體公司的分類體系（例如「新功能 / 修 bug / 技術債 / 維運」）不適用我們。我們的產業有「驗證」、「認證」、「製造導入」這幾個不可壓縮的大筆投入——一塊板子要過 FCC、CE、BSMI 認證的工時，跟修一個 bug 的工時不是同一件事，也不該被合在「品質」桶子裡。 ...
